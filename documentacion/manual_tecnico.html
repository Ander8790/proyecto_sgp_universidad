CAPÍTULO 01
Descripción General del Sistema

El Sistema de Gestión de Pasantías (SGP) es una aplicación web institucional desarrollada para la División de Soporte Técnico del Instituto de Salud Pública del Estado Bolívar. Digitaliza y centraliza la administración de las pasantías: registro de pasantes, control de asistencias, asignaciones, evaluaciones y generación de reportes.

1.1 Objetivos Técnicos
  • Proveer una plataforma web accesible desde red local (LAN) sin dependencia de internet.
  • Implementar un sistema de roles y permisos granulares para SuperAdmin, Admin, Tutor y Pasante.
  • Registrar asistencias mediante Kiosco con PIN personal y validación de período activo.
  • Generar reportes PDF y exportaciones directamente desde el navegador.
  • Auditar todas las acciones relevantes a través de la bitácora del sistema.
  • Notificar en tiempo real al administrador mediante notificaciones de escritorio (Web Notifications API).
1.2 Alcance del Sistema
✅ Incluye
Gestión de pasantes · Control de asistencias · Evaluaciones por competencias · Reportes y analíticas · Días feriados · Backup · Bitácora · Notificaciones de escritorio
❌ Excluye
Integración con sistemas externos · Autenticación LDAP/SSO · App móvil nativa · Facturación · Comunicación por correo electrónico

3.2 Pasos de Instalación
  1. Instalar XAMPP (incluye Apache + PHP + MySQL) o equivalente.
  2. Copiar la carpeta proyecto_sgp/ en C:\xampp\htdocs\.
  3. Habilitar mod_rewrite en httpd.conf si no está activo.
  4. Crear la base de datos proyecto_sgp en MySQL con charset utf8mb4.
  5. Importar el script SQL de database/migrations/.
  6. Ejecutar los seeds en database/seeds/ para datos iniciales.
  7. Verificar app/config/config.php: credenciales de BD.
  8. Acceder a http://localhost/proyecto_sgp/sgp/public/.
3.3 Configuración de Apache (.htaccess)
RewriteEngine On RewriteCond %{REQUEST_FILENAME} !-f RewriteCond %{REQUEST_FILENAME} !-d RewriteRule ^(.*)$ index.php?url=$1 [QSA,L]
Nota: El archivo .htaccess debe estar en public/. Si Apache no tiene AllowOverride All, las rutas amigables no funcionarán.

5.3 Relación Crítica: Pasante → Asistencias
Importante: asistencias.pasante_id almacena el usuarios.id directamente, NO el datos_pasante.id. Para obtener el nombre: JOIN datos_personales dp ON a.pasante_id = dp.usuario_id.
-- JOIN correcto para nombre del pasante en asistencias SELECT a.id, CONCAT(dp.nombres, ' ', dp.apellidos) AS nombre, TIME_FORMAT(a.hora_registro, '%h:%i %p') AS hora FROM asistencias a JOIN datos_personales dp ON a.pasante_id = dp.usuario_id WHERE a.fecha = CURDATE();

6.3 Timeout de Sesión

La sesión expira tras 25 minutos de inactividad (SESSION_TIMEOUT_SECONDS = 1500). El middleware AuthMiddleware::require() verifica esto en cada petición protegida. Al expirar, redirige al login.

6.4 Protección CSRF

Todos los formularios POST incluyen un token CSRF generado por CsrfHelper. El controlador valida el token antes de procesar datos. Las peticiones AJAX de notificaciones incluyen el header X-Requested-With: XMLHttpRequest.

Kiosco: La interfaz de Kiosco usa su propio layout y sesión separada para permitir acceso sin credenciales de usuario. La autenticación ocurre mediante PIN personal del pasante.

CAPÍTULO 14
Kiosco de Asistencias

El Kiosco es una interfaz de pantalla completa sin barra de navegación diseñada para ser ejecutada en una tableta o computadora dedicada en recepción. Los pasantes marcan su asistencia ingresando su PIN.

14.1 Flujo de Marcado
  1. El pasante ingresa los dígitos de su PIN en el teclado numérico en pantalla.
  2. El sistema verifica: PIN correcto → estado pasante activo → período activo → no marcó ya hoy.
  3. Si todo es válido: registra la asistencia con método='Kiosco' y muestra confirmación.
  4. Si alguna validación falla: muestra mensaje específico de error.
14.2 Validaciones del Kiosco
ValidaciónError mostrado
PIN incorrecto o no existe"PIN incorrecto. Intenta nuevamente."
Estado pasante ≠ activo"Tu pasantía no está activa. Contacta al Administrador."
Período no activo/planificado"El período académico actual no está activo. Contacta al Administrador."
Ya marcó hoy"Ya registraste tu asistencia hoy."

CAPÍTULO 21
Backup del Sistema

El módulo de backup permite generar y descargar respaldos completos de la base de datos en formato SQL. Solo accesible para SuperAdmin.

21.1 Proceso de Backup
  1. El SuperAdmin accede al módulo Backup desde el menú lateral.
  2. Selecciona el tipo de respaldo (completo, solo datos, solo estructura).
  3. El sistema genera el archivo SQL mediante mysqldump o consultas PHP.
  4. El archivo se descarga automáticamente con nombre: sgp_backup_YYYYMMDD_HHMMSS.sql.
  5. La acción queda registrada en la bitácora.
📷

23.2 Actualización del Sistema
  1. Hacer backup completo de BD antes de cualquier actualización.
  2. Copiar nuevos archivos al directorio del proyecto (no sobreescribir app/config/config.php).
  3. Ejecutar scripts de migración SQL si hay cambios de esquema.
  4. Limpiar caché del navegador en los equipos cliente.
  5. Verificar funcionamiento en todas las rutas principales.
23.3 Monitoreo de Rendimiento

Para instalaciones con más de 50 usuarios concurrentes, revisar:

  • Índices en asistencias.fecha, asistencias.pasante_id, bitacora.created_at.
  • Activar OPcache en PHP para mejorar tiempo de respuesta.
  • Limitar el historial de bitácora a 6 meses en instalaciones de alto uso.
